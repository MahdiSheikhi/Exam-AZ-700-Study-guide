Virtual Networks (VNets)

Virtual Network Fundamentals

Azure Virtual Network (VNet) is the fundamental building block for your private network in Azure. It enables Azure resources to securely communicate with each other, the internet, and on-premises networks.

Key Concepts

  • Address Space: Custom private IP address space using public and private (RFC 1918) addresses
  • Subnets: Segments of the virtual network address space
  • Regions: VNets are scoped to a single region/location
  • Subscriptions: VNets are scoped to a subscription
# Azure CLI - Create Virtual Network az network vnet create \ --name MyVNet \ --resource-group MyResourceGroup \ --location eastus \ --address-prefix 10.0.0.0/16 \ --subnet-name MySubnet \ --subnet-prefix 10.0.1.0/24

VNet Architecture Patterns

Common VNet architectures include hub-and-spoke, mesh, and hybrid configurations. The diagram above shows a typical hub-and-spoke topology with centralized services.

VNet Design Best Practices

Address Space Planning

  • Use non-overlapping address spaces
  • Reserve address space for future growth
  • Plan for multiple environments (dev, test, prod)
  • Consider on-premises network integration

Subnet Design

  • Use large subnets for flexibility
  • Reserve space for Azure services
  • Plan for subnet delegation
  • Consider security boundaries

Network Security Groups (NSGs)

Network Security Groups (NSGs) are Azure's basic network security feature. They act as a basic firewall, allowing or denying traffic based on 5-tuple rules.

NSG Rule Components

  • Source IP: Source IP address or range
  • Source Port: Source port or range
  • Destination IP: Destination IP address or range
  • Destination Port: Destination port or range
  • Protocol: TCP, UDP, or Any
# Azure CLI - Create NSG and add rules az network nsg create \ --name MyNsg \ --resource-group MyResourceGroup \ --location eastus # Add inbound rule for HTTP az network nsg rule create \ --name AllowHTTP \ --nsg-name MyNsg \ --resource-group MyResourceGroup \ --priority 100 \ --direction Inbound \ --access Allow \ --protocol Tcp \ --source-address-prefix '*' \ --source-port-range '*' \ --destination-address-prefix '*' \ --destination-port-range 80

NSG Rule Priority

Priority 100
Allow HTTPS from Any
Priority 110
Allow HTTP from Any
Priority 4000
Deny All Inbound

Lower numbers = Higher priority. Rules are processed in priority order.

User-Defined Routes (UDRs)

User-Defined Routes (UDRs) allow you to customize traffic routing within your VNet. They override Azure's default routing behavior.

Common UDR Scenarios

  • Force tunneling through NVA
  • Route traffic between spokes via hub
  • Direct internet traffic through firewall
  • Implement custom routing policies
# Azure CLI - Create Route Table and UDR az network route-table create \ --name MyRouteTable \ --resource-group MyResourceGroup \ --location eastus # Add route to force internet traffic through NVA az network route-table route create \ --name ForceInternetTraffic \ --route-table-name MyRouteTable \ --resource-group MyResourceGroup \ --address-prefix 0.0.0.0/0 \ --next-hop-type VirtualAppliance \ --next-hop-ip-address 10.0.0.4

Practice Questions

Virtual Networks

You need to create a VNet with address space 10.0.0.0/16. How many total IP addresses are available?

A) 16,384
B) 32,768
C) 65,536
D) 65,534
Correct! A /16 network provides 65,536 total addresses, but Azure reserves 5 IP addresses per subnet, leaving 65,534 usable addresses.
NSG Rules

You have an NSG with the following rules: Priority 100 (Allow HTTPS), Priority 200 (Allow HTTP), Priority 3000 (Deny All). What happens to HTTPS traffic?

A) Denied by default rule
B) Allowed by priority 100 rule
C) Allowed by priority 200 rule
D) Processed by priority 3000 rule
Correct! NSG rules are processed in priority order (lower numbers first). HTTPS traffic matches the priority 100 rule and is allowed.
User-Defined Routes

You want to force all internet traffic through a Network Virtual Appliance (NVA) with IP 10.0.0.100. What address prefix should you use in your UDR?

A) 10.0.0.0/16
B) 192.168.0.0/16
C) 0.0.0.0/0
D) 255.255.255.255/32
Correct! The 0.0.0.0/0 prefix represents all possible IP addresses (default route), which will catch all internet-bound traffic.
